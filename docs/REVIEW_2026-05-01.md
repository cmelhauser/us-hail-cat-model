# v2.1 Comprehensive Review — PhD-Level / Production-Grade Readiness

**Project:** CONUS Hail Catastrophe Model (us-hail-cat-model)
**Reviewer scope:** README, all `docs/*.md`, every stage script, `run_pipeline.py`, `tests/*`, `requirements.txt`
**Review date:** 2026-05-01
**Reviewed against:** v2.1 hardening release (post-merge, pre-first-execution)

---

## 1. Executive Summary

The v2.1 release is **methodologically sound and well-organized** for a public-data hail hazard model. The architecture is correct (radar-first, sparse event storage, regional EVT pooling, dual analytical/empirical RP products), the documentation is unusually complete for a research repo (12 docs files, plain-language explainer, AI agent instructions, pre-run review), and the test suite is wired into every stage.

The gap between "well-organized v2.1 hardening release" and "PhD-level production-grade" is real but addressable without methodology changes. It falls into six categories:

1. **Project metadata is incomplete** — no LICENSE file, CHANGELOG, CITATION.cff, CONTRIBUTING, or version-pinned environment.
2. **No CI/CD or containerization** — tests must be run manually; no reproducible build environment.
3. **Code engineering hygiene** — grid constants are duplicated across 15 scripts, logging is `print` + manual file write, no type checking, no linting config, no `pyproject.toml`.
4. **Methodology lacks formal uncertainty quantification** — RP maps have no confidence bounds; GPD threshold scoring weights are unjustified; spatial dependence has no diagnostics; σ_perturb calibration method is under-documented.
5. **Documentation has stale and inconsistent metadata** — Python version disagrees across files; `requirements.txt` header still says v2.0; mathematical notation is informal text rather than LaTeX.
6. **No benchmark validation** — outputs are not yet compared against any published RP estimates (Cintineo et al., Allen et al., Wendt & Jirak), and there are no regression / golden-output tests.

The findings below are organized by category with severity tags: **P0** = blocks a defensible run / publication, **P1** = needed for production-grade, **P2** = polish / nice-to-have.

#### A.3 [P1] No environment lockfile

`requirements.txt` uses `>=` constraints. For reproducibility you need either:

- `requirements.lock` (from `pip-tools`: `pip-compile requirements.in -o requirements.lock`), **or**
- `poetry.lock` / `pdm.lock` / `uv.lock`, **or**
- `conda-lock.yml` (likely best given cartopy + cfgrib + eccodes are conda-friendly)

A Stage 04b run that succeeds with `numpy 1.24` may behave differently with `numpy 2.x` (e.g. `np.float_` removed, `np.bool8` removed). Pin everything you actually run with.

#### A.4 [P0] No CI/CD

The pre-run command sequence (`py_compile`, `pytest`, `--dry-run`) is **a CI workflow waiting to happen**. Without CI:

- Test failures only surface when someone manually runs `pytest`.
- PRs can merge with broken stages.
- There's no signal that the documented commands actually still work.

**Recommended `.github/workflows/tests.yml`:**

```yaml
name: tests
on:
  push:
    branches: [main]
  pull_request:
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - name: Install system deps
        run: sudo apt-get update && sudo apt-get install -y libeccodes-dev libgeos-dev proj-bin
      - name: Install Python deps
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest pytest-cov ruff mypy
      - name: Lint
        run: ruff check .
      - name: Type check
        run: mypy scripts/ --ignore-missing-imports
      - name: Compile check
        run: python -m py_compile run_pipeline.py scripts/*.py
      - name: Test
        env:
          OPENBLAS_NUM_THREADS: 1
          PYTEST_DISABLE_PLUGIN_AUTOLOAD: 1
        run: pytest -q tests --cov=scripts --cov-report=xml
      - name: Dry run
        run: python run_pipeline.py --dry-run
      - name: Coverage
        uses: codecov/codecov-action@v4
```

#### B.3 [P1] Mathematical notation is informal

`docs/methodology.md` writes equations as plain text in code fences, e.g.:

```text
MESH75 = 15.096 × SHI^0.206
factor = 1 + α × elevation_km / max(freezing_level_km, ε)
factor = clip(factor, 1.0, 1.25)
```

For PhD-level rigor, use proper LaTeX/MathJax (GitHub supports `$$...$$` math blocks):

```math
\text{MESH75} = 15.096 \cdot \text{SHI}^{0.206}
```

```math
f(z, z_{0\degree C}) = \mathrm{clip}\!\left(1 + \alpha \cdot \frac{z}{\max(z_{0\degree C}, \varepsilon)},\; 1.0,\; 1.25\right)
```

Define every symbol the first time it's used, with units. Write a notation glossary in `methodology.md` §0 listing all symbols.

#### B.4 [P0] Missing uncertainty methodology document

There is no `docs/uncertainty.md`. PhD-level catastrophe models must explicitly document the uncertainty budget:

- **Measurement uncertainty** — radar reflectivity error, SHI integration error, MESH75 calibration residuals (Murillo & Homeyer 2021 reports them — quote)
- **Algorithmic uncertainty** — block-maximum quantization, ERA5 monthly isotherm interpolation error
- **Sampling uncertainty** — 28-year record limits tail estimation; bootstrap 95% CI on RP estimates
- **Model uncertainty** — lognormal-body vs Weibull-body sensitivity; GPD threshold sensitivity (already partially addressed in stage 09 diagnostics — formalize); regional ξ pooling vs cell-specific
- **Stochastic uncertainty** — σ_perturb posterior, translation kernel choice
- **Vulnerability uncertainty** — placeholder caveat (already present)

Each RP map should have a companion uncertainty raster. Suggested stage 09 output addition: `rp_XXXXXyr_hail_q05.tif` and `rp_XXXXXyr_hail_q95.tif` from non-parametric bootstrap on cell-level annual maxima with regional ξ resampled.

#### C.2 [P1] `print` + manual file write is not a logger

Every stage has its own `log()` function that does `print` + manual file append. This:

- can't be filtered by level (no DEBUG / INFO / WARN / ERROR distinction)
- can't emit structured (JSON) logs
- can't be redirected by callers
- duplicates code 15 times

**Fix:** Use `logging` with a single configuration helper. `scripts/_logging.py`:

```python
import logging
import sys
from pathlib import Path

def get_logger(stage_name: str, log_dir: Path | None = None) -> logging.Logger:
    """Configure a stage logger. Writes to logs/<stage_name>.log + stderr."""
    logger = logging.getLogger(stage_name)
    if logger.handlers:
        return logger  # already configured
    logger.setLevel(logging.INFO)
    fmt = logging.Formatter(
        "%(asctime)s [%(levelname)s] [%(name)s] %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )
    sh = logging.StreamHandler(sys.stderr)
    sh.setFormatter(fmt)
    logger.addHandler(sh)
    if log_dir is not None:
        log_dir.mkdir(parents=True, exist_ok=True)
        fh = logging.FileHandler(log_dir / f"{stage_name}.log", mode="a")
        fh.setFormatter(fmt)
        logger.addHandler(fh)
    return logger
```

Each stage:

```python
from _logging import get_logger
from _config import LOG_ROOT
log = get_logger("05_apply_mesh_bias_correction", LOG_ROOT).info
```

### D. Testing & Validation Rigor

#### D.1 [P0] No integration test

All tests are pure unit (helper functions on hand-crafted arrays). There is no end-to-end smoke test that runs the full pipeline on a tiny synthetic dataset and verifies the output schema.

**Recommended:** `tests/integration/test_smoke_synthetic.py` that:

1. Writes a 5×5 synthetic raster grid for 10 days
2. Runs stages 05 → 13 in sequence with `--skip-ml` and `--n-years 50`
3. Asserts every expected output file exists
4. Asserts RP map values are non-negative, finite, and monotonic with RP

This catches stage-boundary breakage that unit tests miss.

#### D.2 [P0] No regression / golden-output tests

Once you do a full run, you should freeze the resulting RP100 map, p_occ_1.00in map, and event count for posterity. A `tests/regression/test_golden_outputs.py` runs against fixed input data and compares to a checksum or array hash. Any change in numerics requires explicit acknowledgment.

#### D.3 [P1] No coverage measurement

Add `pytest-cov` and require a floor (e.g. 80%) in CI. Some stage helpers (e.g. `compute_mrl_and_threshold`, `physically_coherent_merge`, `translate_sparse`) are critical and should be at 100%.

#### D.4 [P1] No property-based tests for sparse stochastic invariants

Stage 13 has invariants that property testing (`hypothesis`) can verify across thousands of random inputs:

- After translation, all (rows, cols) are in `[0, NROWS)` × `[0, NCOLS)`
- After intensity scaling, all vals are in `[0, MAX_HAIL_MM]`
- The active-cell index `lookup` is a bijection on visited cells
- `update_sparse_max(row, ...)` is idempotent (`f(f(x)) == f(x)` when called with same args)
- `compute_empirical_rps` produces values monotone-decreasing in rank

Example:

```python
from hypothesis import given, strategies as st

@given(
    rows=st.lists(st.integers(0, 519), min_size=1, max_size=100),
    cols=st.lists(st.integers(0, 1179), min_size=1, max_size=100),
    sigma=st.floats(min_value=0.5, max_value=20.0),
)
def test_sparse_translation_stays_in_domain(rows, cols, sigma):
    s = load_stage("13_generate_stochastic_catalog.py")
    rng = np.random.default_rng(0)
    rr = np.array(rows[: min(len(rows), len(cols))], dtype=np.int32)
    cc = np.array(cols[: len(rr)], dtype=np.int32)
    out_r, out_c, *_ = s.translate_sparse(rr, cc, rng, sigma_cells=sigma)
    assert np.all((out_r >= 0) & (out_r < s.NROWS))
    assert np.all((out_c >= 0) & (out_c < s.NCOLS))
```

### E. Scientific Methodology Rigor (PhD-Level)

This is where the largest gaps are for "PhD-level" claim. The methodology is defensible but not yet rigorous in the formal-uncertainty sense.

#### E.1 [P0] No uncertainty bounds on RP estimates

Currently each cell gets a point estimate `rp_100yr_hail.tif`. PhD-level requires:

- **Bootstrap CIs** on cell-level annual maxima (block bootstrap by year to preserve temporal dependence)
- **Regional ξ uncertainty** propagated through cell-specific σ
- **Output:** companion `rp_100yr_hail_q05.tif` and `rp_100yr_hail_q95.tif` rasters

Implementation sketch in stage 09:

```python
# After fitting regional ξ, do B=200 bootstrap iterations:
xi_boot = np.zeros((B, n_regions))
for b in range(B):
    for reg in range(n_regions):
        exc_b = rng.choice(exceedances[reg], size=len(exceedances[reg]), replace=True)
        # Refit GPD via L-moments
        xi_boot[b, reg] = lmoments_xi(exc_b)
# Propagate to RP maps via Monte Carlo
rp_samples = np.zeros((B, NROWS, NCOLS), dtype=np.float32)  # only if memory allows
# OR: store p05 / p95 cell-wise in two passes
```

This adds runtime (~10× current Stage 09) but is the single biggest methodology upgrade.

#### E.2 [P0] GPD threshold scoring weights are arbitrary

Stage 09 `compute_mrl_and_threshold` scores candidate thresholds with:

```python
score = ks + mrl_score + stability + count_penalty
```

These four terms have different units and scales. A small `count_penalty` might dominate or be ignored depending on regional sample size. PhD-level requires:

- Documented weight rationale (or document that all components are normalized to `[0, 1]` first)
- Sensitivity analysis: re-pick threshold under {1:1:1:1, 2:1:1:1, 1:2:1:1, ...} and report how often the selection changes

A defensible alternative: pick threshold by maximizing **likelihood-ratio-test stability** (Coles 2001 §4.3.4) plus a minimum-exceedances floor.

#### E.3 [P0] σ_perturb calibration method is under-documented

Stage 13 docstring says σ is "calibrated from empirical inter-annual intensity variance." The actual function `calibrate_sigma()` was not visible in my review pass. Whatever the method is, `docs/methodology.md` §13 needs:

```math
\hat{\sigma}_{\text{perturb}} = \sqrt{\mathrm{Var}\!\left[\ln(\text{annual peak hail across active cells})\right]}
```

with the exact estimator (sample variance? trimmed? region-stratified?), what's resampled vs treated as fixed, and whether σ is global or region-specific. If global, justify that hail intensity variability is region-invariant (it isn't really — likely should be regional).

#### E.4 [P1] Event independence assumption is not tested

Stage 13 draws `N_events ~ Poisson(λ)`. This assumes events are temporally independent. Hail is famously clustered (active synoptic patterns produce multiple events). Diagnostic to add to Stage 08:

- **Index of dispersion** of annual event counts: `var/mean`. If »1, Poisson under-disperses tail.
- **Allan factor** at varying scales
- **Negative binomial fit** as alternative; report dispersion parameter

If significantly over-dispersed, the methodology should switch to NegBin in Stage 13 (one-line code change but documented model assumption).

#### E.5 [P1] Spatial dependence has no diagnostics

`docs/methodology.md` §10 acknowledges this but only as a future-work mention. PhD-level requires at minimum:

- **Empirical extremogram** at lags up to 500 km using top-quantile threshold (Davis & Mikosch 2009)
- **Pairwise tail dependence** χ(u) for u → 1 across representative cell pairs
- **Aggregate-vs-marginal comparison**: what fraction of stochastic 100-yr aggregate loss vs sum of cell-level 100-yr does the model produce? If they're equal, you have artificial independence. If they differ by a factor of 2+, document.

These are diagnostic-only for v2.1 but should be in `docs/figures/analysis/` and discussed in `docs/methodology.md` §10.

#### E.6 [P1] No source-homogeneity check after Stage 05

Stage 05 calibrates GridRad to MYRORSS using 2005–2011 overlap. After calibration, do a 2-sample Kolmogorov-Smirnov test on the joint distribution of corrected MESH75 between (MYRORSS-only days) and (GridRad-only days) within the overlap. If KS rejects, the calibration is incomplete. Save to `data/historical/validation/source_homogeneity.csv`.

#### E.7 [P1] Vulnerability MDR parameters need derivation trace

`scripts/14_build_vulnerability.py` has μ_v / σ_v values as comments like `"ln(~20 mm) — damage onset ~20 mm"`. These appear hand-tuned, not derived. PhD-level: append a `docs/vulnerability_derivation.md` showing how each μ_v / σ_v was obtained from Brown et al. 2015 + IBHS — even if approximate, document the trace, the assumed roof age distribution, and the eyeballed envelope.

If they truly are placeholders, mark them as such *in the script* via a class-level flag (`PARAMS_ARE_PLACEHOLDER = True`) and have stage 14 print a banner unless the user opts in via `--accept-placeholder-vulnerability`.

#### E.8 [P1] Topographic correction coefficient (0.25) is uncited

`scripts/12_apply_conus_mask.py: compute_topo_factor` uses:

```python
factor = 1.0 + 0.25 * (elev_km / fl)
```

The 0.25 has no citation in `docs/literature_review.md` §8. Either:

- Cite a source (Rasmussen & Heymsfield 1987 melt model can give a reasonable bound)
- Document it as "tuned to give Front Range / Great Plains ratios consistent with Cintineo et al. 2012"
- Or treat as a calibration parameter with documented sensitivity (B.5)

#### E.9 [P2] Mann-Kendall trend test should be implemented as a diagnostic stage

Literature review §10 recommends it. It's ~20 lines of code. Add as Stage 06b or as a routine in Stage 15 figure rendering. Saves a per-cell trend map and CONUS-aggregated trend statistic (with autocorrelation correction via Hamed-Rao or Yue-Wang).

#### E.10 [P2] Stage 09 default 6 regions deserves cited basis

K-means with k=6 is implementation-driven, not science-driven. Compare against Allen et al. 2015 hail regions (their Fig 5 has ~5 climatological clusters), or cite the silhouette-score / gap-statistic optimum k from a sensitivity test. If the optimum k differs from 6, change the default.

### F. Reproducibility & Operational Readiness

#### F.1 [P0] No checksums on input data

A run that downloads MYRORSS day X today vs day X next year may produce different outputs if NOAA reprocesses files. Stage 01/02/03 should compute SHA-256 of every downloaded file and write to `data/historical/<source>/checksums.csv`. The run manifest references this checksum file.

#### F.2 [P1] No checkpoint within a stage

Stage 01 (MYRORSS download) is multi-day. If it crashes at year 2007, current behavior is "skip files that already exist." Better: write `<stage>_progress.json` after each year so a resume knows which year to pick up.

#### F.3 [P1] No data versioning (DVC)

Once data files exist, version them with DVC against an S3 / minIO backend. Without this, "reproduce a paper's figure 3" requires the exact data files, which are gitignored.

#### F.4 [P2] No idempotency contract

Each stage should specify in its docstring: "Running this stage twice with the same inputs produces identical outputs." Verify in tests (run twice, assert outputs are byte-identical). Most stages probably are; Stage 09 might not be due to KMeans non-determinism — check `KMeans(random_state=42)` is set everywhere.
